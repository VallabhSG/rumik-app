1. Client-Side Architecture

The React Native client is organised in four layers. The UI layer renders components. The hook layer bridges context and components. The context layer holds shared state. The service layer contains all business logic and has no React dependencies.

Key Algorithms

Rollout Bucketing (rollout.ts)

Uses DJB2 hash of installId mapped to 0–99. Device is eligible if hash % 100 < rollout_percentage. Deterministic — same device always gets same result.

Semver Enforcement (semver.ts)

Compares native app version against release's min_native_version / max_native_version. Uses integer-tuple comparison (major · minor · patch).

PerfTracker Flush Guard

Enforces MIN_FLUSH_INTERVAL = 10s between flushes. On failed flush, restores queue so no metrics are lost. Auto-flushes on stop().

ErrorReporter Stack Parsing

Regex parses at funcName (file:line:col) format. Filters node_modules frames. Takes top 20 frames. Gracefully handles missing error.stack.

2. Server API Design

All endpoints follow a consistent response envelope: { success, data, error?, meta? }. Authentication uses Bearer token via Authorization header. Two server instances are deployed — Staging and Production — each with an independent SQLite database. Release and crash-rate records are scoped by the channel field (staging | production).

3. Database Schema

SQLite database with 14 tables, 8 indexes for query performance. All IDs are UUID v4. Timestamps are ISO 8601 strings.

4. Alert Engine — Internal Flow

Runs every 5 minutes via setInterval in the Express server. Evaluates all enabled rules and fires Slack-compatible webhooks when thresholds are breached and the cooldown window has passed.

6. Remote Config Resolution

The GET /api/config endpoint resolves all config types for a specific device context in a single request, applying targeting rules to each entry. Staging builds point to the staging server; production builds point to the production server — controlled by the OTA_SERVER_URL environment variable baked in at EAS build time.

Test Isolation Pattern (OTA Server)

Each route test file creates a fresh in-memory SQLite DB via createTestDb() and mocks the db.js module before any route imports. This ensures full isolation — no shared state between test files even when running in parallel.
